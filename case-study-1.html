EdTech UX Research Product Design

Building a better question experience

How I redesigned Top Hat's question creation flow to reduce cognitive load, decrease support tickets by 40%, and increase completion rates by 25%.

Role Senior Product Designer
Timeline 4 months
Team 1 PM, 3 Engineers, 1 Designer
Platform Web Application

Hero image: Final design mockup

The challenge

Professors struggled to create questions efficiently, leading to frustration, incomplete assessments, and a high volume of support tickets.

The solution

A redesigned question builder with progressive disclosure, inline validation, contextual help, and a streamlined workflow.

The impact

40% reduction in support tickets, 25% increase in question completion rate, and significantly improved user satisfaction.

Context

Understanding the problem space

Top Hat is an EdTech platform that helps professors engage students through interactive classroom experiences. One of its core features is the ability for educators to create various question types for lectures, assignments, and assessments.

When I joined the team, the question creation tool had grown organically over several years. While it offered powerful functionality, the interface had become cluttered and confusing. New users often abandoned the process, and even experienced professors reported frustration with complex question types.

Research

Discovering user pain points

I began by conducting extensive research to understand where users were struggling and why. This involved analyzing support tickets, reviewing session recordings, and conducting interviews with professors.

Research methods

12 User Interviews
50+ Session Recordings
200 Support Ticket Analysis
Analytics Deep Dive

Key findings

1
Overwhelming initial interface

Users were presented with all options at once, regardless of the question type they were creating. This created decision paralysis.

2
Unclear error states

When something went wrong, users received vague error messages that didn't explain what to fix or how.

3
Missing contextual guidance

Advanced features lacked explanation, leaving users unsure when or why to use them.

"I just want to create a simple multiple choice question. Why do I need to see all these advanced settings?"

Biology Professor First-year user

"I spent 20 minutes trying to figure out why my question wouldn't save. The error message just said 'invalid input.'"

Chemistry Professor 3-year user
Design

Crafting the solution

Armed with research insights, I developed a design strategy centered on three principles: progressive disclosure, clear feedback, and contextual guidance.

Design principles

1. Progressive disclosure

Show only what's needed at each step. Advanced options are available but hidden until relevant.

Before

All 15+ options visible at once

After

3 core options, expandable sections for advanced

2. Inline validation & clear feedback

Validate inputs as users type. Provide specific, actionable error messages.

Inline validation example showing real-time feedback

3. Contextual help & guidance

Tooltips and inline help explain features in context, reducing the need for documentation.

Contextual tooltip showing feature explanation

Iteration

Testing & refining

I conducted three rounds of usability testing with professors of varying experience levels. Each round informed improvements before development began.

Round 1
Concept Validation

Tested core navigation and progressive disclosure concept

Round 2
Interaction Testing

Refined error states and validation timing

Round 3
Final Validation

Confirmed improvements with high-fidelity prototype

Key iteration: Error message placement

Initial testing revealed that error messages at the top of the form went unnoticed. I moved them inline with the problematic field, which increased error resolution rate from 60% to 95%.

Reflection

What I learned

Key takeaways

  • Progressive disclosure is powerful, but requires careful testing to get right
  • Error messages are a critical part of the user experience, not an afterthought
  • Contextual help reduces cognitive load more effectively than documentation
  • Involving users throughout the process catches issues early

What I'd do differently

  • Conduct more interviews with new users specifically
  • Create a more robust component library earlier in the process
  • Set up analytics tracking before launch to capture baseline metrics
